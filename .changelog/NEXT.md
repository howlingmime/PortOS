## Added

- **`/brain/links` â€” per-repo malware/risk scan button.** New "Scan" action (ShieldCheck icon) sits next to Open / Pull on every cloned-status GitHub repo. Clicking it queues a CoS user task that runs `/do:scan` against the clone's `localPath` â€” the bundled slashdo command is a hardened read-only audit (no scanned code is executed, no fetched URLs are followed, hostile-until-proven-safe). The agent writes its full markdown report to `~/.claude/scans/<basename>-<date>.md` and the user is told to track progress in CoS Tasks. Backed by a new `POST /api/brain/links/:id/scan` endpoint that loads the bundled slashdo body via `loadSlashdoCommand('scan')` and bakes the repo path into the task context as `SCAN_DIR`. Pre-flight checks reject non-cloned links, missing `localPath`, vanished folders, and duplicate in-flight scans for the same repo. Files: `server/routes/brain.js`, `client/src/services/apiBrain.js`, `client/src/components/brain/tabs/LinksTab.jsx`. The slashdo submodule is also bumped from `a38ee04` to `50011ef` (picks up `do:pr-better` and the v2.12.0 release).

## Changed

- **`loadSlashdoFile()` now caches resolved command bodies** (server/lib/fileUtils.js) â€” slashdo command markdown + their `!cat ~/.claude/lib/...` includes are static within a server lifetime (submodule updates require restart anyway), so the per-call ~26KB read + N-include resolution is now memoized in a process-lifetime `Map` keyed by `${commandName}::${stripFrontmatter}`. Eliminates redundant disk I/O for callers like the new `/brain/links/:id/scan` endpoint and the existing `/api/cos/tasks/slashdo` dispatch.

- **`docs/DEPS.md` â€” dependency audit reference.** New living doc that records every third-party dependency in PortOS, its tier classification (1 acceptable / 2 audited / 3 removable), and per-package rationale for being kept. Generated by `/do:depfree` runs so future audits can cross-reference prior verdicts and re-audit triggers (e.g. "if `pm2`/`needle` drops `sax`, revisit"). First pass (2026-04-28, default mode): 30 deps audited, 0 removals â€” `kokoro-js` kept (only pure-JS in-process TTS, replacement requires Python subprocess + pooling), `sax` flagged REMOVE then downgraded to KEEP (transitive) because `pm2 â†’ needle@2.4.0 â†’ sax@1.4.4` already pulls the same major version (direct removal saves zero supply chain attack surface). - **One-click "Enable HTTPS" button on the Instances tab.** The Tailnet DNS & trusted HTTPS explainer used to instruct users to drop into a shell and run `npm run setup:cert`. New button on the same banner runs the same flow at runtime via a new `POST /api/instances/provision-cert` endpoint, which calls `tailscale cert` for the local MagicDNS hostname, writes `data/certs/{cert,key}.pem`, and updates `meta.json`. Failure modes (Tailscale not installed, not running, no MagicDNS hostname, HTTPS Certificates feature disabled in the tailnet admin) surface as toasts with actionable guidance. Because the HTTPS listener type is decided at server boot in `lib/tailscale-https.js`, the response includes `requiresRestart: true` whenever PortOS is currently running over HTTP â€” the UI then prompts the user to restart so the new cert actually takes effect; if HTTPS was already running, the existing `certRenewer` hot-swap path picks up the renewed cert without a restart. The button is disabled until MagicDNS is detected, mirroring what the underlying CLI requires. Files: `server/services/certProvisioner.js` (new), `server/routes/instances.js`, `client/src/services/apiSystem.js`, `client/src/pages/Instances.jsx`.
- **Image Gen â€” third backend: Codex CLI `$imagegen`.** Image Gen can now route generation through the user's locally-installed Codex CLI built-in `image_gen` tool â€” invoked headlessly with `codex exec --skip-git-repo-check --sandbox workspace-write '$imagegen <prompt>'` so it uses the user's logged-in Codex session (no `OPENAI_API_KEY` required). New provider at `server/services/imageGen/codex.js` mirrors `local.js` (single-job model, SSE progress events) but harvests the resulting PNG from `~/.codex/generated_images/<session-id>/ig_*.png` after parsing the session id out of the codex launch banner â€” codex's stdout has no machine-readable image path, so banner-parse + dir-poll is the contract. Failure modes are surfaced clearly via `imageGenEvents`: empty session dir â†’ "your Codex account may not allow image_gen", missing session id â†’ "output format may have changed", non-zero exit â†’ tail of stderr. Codex usage is gated by an explicit `Enable Codex Imagegen` setting because not every Codex plan exposes the `image_gen` tool â€” when disabled the option disappears from settings tile, image-gen page chip, voice tool dispatch, and palette gating up front.
- **Image Gen â€” per-render backend chip.** New chip strip on the Image Gen page lets the user flip between Local / External / Codex per-render without changing the saved default. Only viable backends (External URL set, Local Python set, Codex enabled) appear; the strip hides itself when only one backend is configured. Selection is sent as `mode:` in `POST /api/image-gen/generate` and the dispatcher honors the override. Codex mode also hides backend-irrelevant knobs (steps, seed, guidance/CFG, LoRAs, quantize) since the built-in `image_gen` tool decides those internally. `GET /api/image-gen/status?mode=codex` lets the page probe a non-default backend before saving.

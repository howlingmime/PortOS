# Next Release

## Added

- **3D Chief of Staff avatars are draggable to re-orient.** All five 3D avatar styles (Cyber, Sigil, Esoteric, Nexus, Muse) now respond to typical mouse/touch drag controls — click-and-drag rotates the avatar around its center, with damping for a smooth feel. Pan and zoom are intentionally disabled so the framing stays consistent on the small avatar tile. Wrapper divs use `cursor-grab`/`active:cursor-grabbing` and `touch-none` so the affordance is visible and touch drags don't fight page scroll. Implemented via drei's `OrbitControls` inside each avatar's `<Scene>`. Files: `client/src/components/cos/CyberCoSAvatar.jsx`, `client/src/components/cos/SigilCoSAvatar.jsx`, `client/src/components/cos/EsotericCoSAvatar.jsx`, `client/src/components/cos/NexusCoSAvatar.jsx`, `client/src/components/cos/MuseCoSAvatar.jsx`.
- **Live AI operation status — toasts + server logs for every model call.** The user had no idea when a feature like Digital Twin → Taste → "Generate Summary" was actually doing work versus hung; the button just animated and silently waited up to a minute while LM Studio loaded a model on demand. New `server/services/aiStatusEvents.js` exports an `EventEmitter` + `startAIOp({ op, label, providerId, providerName, model })` helper that emits phase-tagged events (`start`, `model:loading`, `model:loaded`, `complete`, `error`) with provider/model/op metadata. `server/services/socket.js` bridges the emitter onto a new `ai:status` Socket.IO channel broadcast to all clients. `server/lib/aiProvider.js` — `callProviderAISimple()` now starts a status op for every API call (logs always; toasts when the caller passes `op`/`opLabel`, plus model-loading/error always), and the LM Studio auto-load path emits `model:loading` ("Loading openai/gpt-oss-20b into LM Studio…") + `model:loaded` ("openai/gpt-oss-20b loaded (4.2s)") so the user sees what's happening instead of a frozen button. `server/services/taste-questionnaire.js` passes labeled ops for the three taste AI calls (section summary, overall summary, deeper personalized question) so toasts read "Generating Movies & Film taste summary…" rather than a generic provider name. New `client/src/hooks/useAIStatusNotifications.js` subscribes to `ai:status`, renders one mutating toast per op id (`toast.loading` → `toast.success`/`toast.error` with the same id so phases update the same toast), and is mounted globally in `client/src/components/Layout.jsx`. Silent (uninstrumented) ops only toast when something user-visible happens (model load, error, or call exceeds 2.5s) so background calls don't spam the UI; explicit user-triggered ops toast immediately. Server logs are single-line emoji-prefixed with op slug, phase, message, provider, model, and elapsed time per CLAUDE.md style. Files: `server/services/aiStatusEvents.js` (new), `server/lib/aiProvider.js`, `server/services/socket.js`, `server/services/taste-questionnaire.js`, `client/src/hooks/useAIStatusNotifications.js` (new), `client/src/components/Layout.jsx`.

## Fixed

- **Taste summaries render as markdown and appear inline immediately after generation.** The Digital Twin → Taste tab had two regressions in the summary flow: (1) after clicking "Generate Summary" on the section-complete view, the button returned to its enabled state with no visible change — the summary was saved but not rendered until the user navigated away and back into the section's Review view, and (2) every place the summary did show (review-mode AI Summary card, expandable summary on the dashboard grid, "Unified Taste Profile" overall summary) used a `whitespace-pre-wrap` `<div>` so the LLM-generated markdown displayed as raw `### Movies & Film Profile`, `**Core Preferences:**`, `- Item 1` etc. instead of styled headings, bold text, and bullets. Now: the section-complete view shows the freshly-generated summary inline in a violet-bordered card the moment `loadProfile()` returns, and all three summary surfaces use the existing `MarkdownOutput` component (which already powers CoS, autobiography, etc.) so headings/bold/italics/bullets/blockquotes render properly. File: `client/src/components/digital-twin/tabs/TasteTab.jsx`.

- **Voice agent no longer feedback-loops on laptop speakers.** When the user ran the voice widget without headphones, TTS played through the laptop speakers, the built-in mic picked it up, STT transcribed the bot's own voice as a "user" turn, the bot replied to itself, and the loop repeated until the user manually stopped it. The existing substring-based echo gate in `voiceClient.js` was too brittle — STT mistranscribes TTS just enough that the heard text wasn't a clean substring of any spoken sentence, so the gate let the echo through. Replaced with a length-gate + trigram-overlap algorithm: utterances under 4 tokens always pass (so short barge-ins like "wait" / "stop" still interrupt the bot even if those words appear in TTS), and longer utterances are flagged as echo when they share ≥ 2 trigrams (3-word windows) with a recently-spoken sentence OR are a clean substring. Same algorithm runs server-side in a new `server/services/voice/echo.js` (per-socket `state.recentTts` ring) so continuous-mode whisper transcripts are also gated, not just Web Speech. Also added device-route detection: when the active mic shares a `groupId` with an audiooutput device AND the label looks headset-y (AirPods, Jabra, etc.), the gate is bypassed entirely (no acoustic echo path on real headphones); a `devicechange` listener re-evaluates on plug/unplug. Default-tight on uncertainty — false-headset would re-open the loop, false-built-in just makes barge-in slightly less twitchy. Files: `server/services/voice/echo.js` (new), `server/services/voice/echo.test.js` (new, 16 tests), `server/services/voice/pipeline.js`, `server/sockets/voice.js`, `client/src/services/voiceClient.js`.
- **Voice agent now navigates to the daily log via `ui_navigate`.** Three related issues caused "take me to the daily log" to fail or behave unexpectedly. (1) The `dailylog` intent regex used `\bdaily ?log\b`, which doesn't match STT typos like "daily login" / "daily logs" — so `daily_log_open` was filtered out of the offered toolset, the LLM only had `ui_navigate`, but `ui_navigate`'s description told it NOT to use that tool for daily log, leaving the model to fabricate a refusal ("Daily log isn't a valid page"). Loosened to `daily ?logi?n?s?` to absorb the drift. (2) Removed the contradictory "EXCEPT Daily Log (use daily_log_open)" carve-out from `ui_navigate`'s description and from the system prompt; the nav resolver already maps "daily log" / "journal" → `/brain/daily-log` correctly. The new rule: plain "go to / open / take me to my daily log" → `ui_navigate`; only pick `daily_log_open` when the user clearly wants to write/dictate ("start", "new", "entry", "make", "dictate"). (3) Extended `detectNarrationWithoutCall` to also catch the second narrate-without-call shape — the LLM emitting a literal tool name in its prose (e.g. `ui_navigate page="daily_log"`) without actually invoking it. The regex is now built dynamically from `getAllToolNames()` so it can't drift behind new tools. Files: `server/services/voice/tools.js`, `server/services/voice/pipeline.js`.
- **Reverting a peer to plain IP (after enabling Tailnet DNS) is now sticky.** When a user cleared a peer's `host` on the Instances page to switch the federation hop back to `http://<ip>`, the next inbound `peers/announce` from that peer would silently re-adopt the DNS name — the only safeguard was "is `existing.host` empty?", which couldn't distinguish "user explicitly cleared" from "never set yet." Added a `hostManual` flag on each peer record: latched to `true` whenever the user sets or clears the host via `PUT /api/instances/peers/:id`, and respected by `handleAnnounce` (auto-adoption is gated on `!existing.host && !existing.hostManual`). Fresh peers and peers that were auto-learned still receive automatic host updates from announces; only explicit user intervention pins the choice. Also added a one-click `use IP only` inline action on each peer's host editor and a discoverable banner at the top of the Instances page that summarizes Tailnet DNS state, surfaces the local instance's MagicDNS name in the SelfCard, and walks through the three-step setup (MagicDNS admin toggle → `npm run setup:cert` → click `use <host>` per peer). Files: `server/services/instances.js`, `server/services/instances.test.js`, `client/src/pages/Instances.jsx`.
- **LM Studio auto-loads a model on "No models loaded" 400.** Any caller of `callProviderAISimple` (taste questionnaire, identity, autobiography, insights, jiraReports, goalCheckIn) that hit a cold LM Studio used to error out with `Provider returned 400: { ... "No models loaded" ... }`. Now on that specific error pattern, the helper lists LM Studio's downloaded LLMs via `/api/v0/models`, picks one (preferring `provider.defaultModel`, then any id in `provider.models` that's downloaded, else the first downloaded LLM), loads it via `POST /api/v1/models/load`, and retries the chat-completion. Mirrors the existing pattern in `services/memoryClassifier.js` but driven from the provider config so every API caller benefits. File: `server/lib/aiProvider.js`.
- **Stop showing the same error toast 3-4 times.** A single failing API call could surface its error message multiple times because three independent channels all toasted the same payload: `apiCore.request` toasts on every non-OK response, the server emits both `io.emit('error:occurred')` (broadcast) and `errorEvents → broadcastToErrors('error:notified')` (subscriber) for the same error, and `useErrorNotifications` listened to *both* socket events with the same handler. Two changes: (1) `useErrorNotifications` now subscribes to the targeted `error:notified` channel only — `error:occurred` was the duplicate broadcast leg — and (2) the Toast layer added a content-based dedup window (1.5s) so any duplicate from any source is silently dropped. Explicit-`id` toasts (loading→success swaps) skip dedup so progress toasts still update in place. Files: `client/src/components/ui/Toast.jsx`, `client/src/hooks/useErrorNotifications.js`.
- **Media Gen Video defaults to the smallest model.** The macOS default was LTX-2 Unified (~42 GB). New default is LTX-2.3 Distilled Q4 (~22 GB) so a fresh install fits on smaller machines without a manual override. File: `server/services/videoGen/local.js`.
- **Taste questionnaire falls back to a configured API provider.** The Digital Twin → Taste tab was auto-selecting the first enabled provider for `Generate Summary` / `Go Deeper`. When that happened to be a CLI provider (Claude Code, Codex, Gemini CLI), the request failed with a generic "requires an API-based provider" toast — three times, because both the API client and two socket subscribers re-toasted the same error. Now (1) the section provider picker filters to API-type providers only, (2) the server resolves any usable API provider when the requested one isn't compatible (requested → active → first enabled API) via a new `resolveAPIProvider` helper in `server/lib/aiProvider.js`, (3) the section-complete view shows an inline link to AI Providers when no compatible provider is configured instead of leaving the user with a wall of red toasts and no direction, and (4) the redundant component-level `toast.error` calls were removed (apiCore already toasts on non-OK responses). Files: `server/lib/aiProvider.js`, `server/services/taste-questionnaire.js`, `client/src/components/digital-twin/tabs/TasteTab.jsx`.

## Changed

- **Send to Video carries the source image's prompts.** Clicking "Send to Video" on an Image Gen result, gallery card, or lightbox — or the same action from Media History — now pre-populates the Video Gen prompt and negative prompt from the source image's metadata. Previously only the source image filename was passed, leaving the user to retype or paste the prompt to keep the same scene direction. The flow piggybacks on the existing `?sourceImageFile=` query-param hand-off: `?prompt=` and `?negativePrompt=` are added when the source has them, and `VideoGen.jsx` reads both on first render and re-syncs on subsequent navigations (mirroring the existing source-image effect). Files: `client/src/pages/ImageGen.jsx`, `client/src/pages/MediaHistory.jsx`, `client/src/pages/VideoGen.jsx`.

## Fixed

- **Voice mode works on Windows.** The voice setup script and TTS engine now run correctly on Windows: `bootstrap.js` invokes `pwsh` with a PowerShell setup script (`scripts/setup-voice.ps1`) instead of the macOS-only bash/Homebrew script; `which` uses `where` on Windows; `verifyBinaries` checks the local piper install path first (skipping a wasteful PATH subprocess when the binary is present); `tts-piper.js` writes piper output to a temp file on Windows to avoid CR+LF corruption of binary WAV data from text-mode stdout. Shared platform constants (`IS_WIN`, `PIPER_BIN_NAME`) extracted to `config.js`.

# Next Release

## Changed

- **PortOS browser launches headed by default; setup/update auto-opens the dashboard in it.** The CDP-managed Chrome instance (`portos-browser`) used to default to `--headless=new`, which made sense for pure CDP automation but meant a fresh `./setup.sh` left no visible browser window even though the user expected to land on the dashboard. Default flipped to headed (`browser/server.js`); `data/browser-config.json` can still set `headless: true` to opt back in for headless workflows. New `scripts/open-ui-in-browser.js` polls `/api/system/health` and `/api/browser/health` (90s API timeout, separate 15s browser/CDP timeout, 500ms interval) and then POSTs to `/api/browser/navigate` with the dashboard URL — `http://localhost:5553` (loopback HTTP mirror) when HTTPS is on so the first paint doesn't trip a cert warning, otherwise `http://localhost:5555`. Hooked into `setup.sh` (after `npm start` returns) and `update.sh` / `update.ps1` (after `npm run pm2:restart`); fail-soft so the parent script's exit code is unaffected by Chrome being unreachable. Files: `browser/server.js`, `scripts/open-ui-in-browser.js` (new), `setup.sh`, `update.sh`, `update.ps1`.

- **`npm run setup` is now the all-in-one install command.** Previously `setup` only ran `setup-data + setup-db + setup-browser` and `install:all` was the longer chain that ended with `&& npm run setup`. That meant a user typing `npm run setup` got an under-prepared workspace (no dependencies installed). Restructured: `setup` now does the full chain (submodules + workspace deps + esbuild postinstall + node-pty rebuild + data/db/browser setup) and `install:all` is a backward-compat alias that calls `setup`. `setup.sh` switched its primary call from `npm run install:all` to `npm run setup`. To keep `update.sh` / `update.ps1` from doing every install twice (their existing `safe_install` retry block + the new `npm run setup` superset), the post-install `npm run setup` call in both scripts was replaced with the three data-side scripts directly (`setup-data` + `setup-db` + `setup-browser`), preserving the resilient ENOTEMPTY retry while avoiding redundant npm installs. The auto-open script's API readiness timeout was also bumped from 30s → 90s (with a separate 15s timeout for the browser-CDP poll) since first-boot can exceed 30s on slow disks. Files: `package.json`, `setup.sh`, `update.sh`, `update.ps1`, `scripts/open-ui-in-browser.js`.

- **Death Clock now multi-color in every surface (DRY)** — extracted the colored 7-unit countdown from `CompactCountdown` in MeatSpace's Overview tab into a shared `DeathClockCountdown` component used by the Dashboard widget, the Ambient/fullscreen view, and MeatSpace itself. The values are now colored consistently (years/months/weeks/days/hours/minutes/seconds → blue/purple/teal/green/yellow/orange/red) with a `size` prop ('sm' for the dashboard widget, 'md' for the MeatSpace hero, 'lg' for Ambient) so each surface picks its own type scale without duplicating the unit/color list. Adds weeks to the Dashboard widget and Ambient view (was 6 units, now 7) to match the MeatSpace layout. Files: `client/src/components/DeathClockCountdown.jsx` (new), `client/src/components/DeathClockWidget.jsx` (use shared), `client/src/pages/Ambient.jsx` (use shared), `client/src/components/meatspace/tabs/OverviewTab.jsx` (use shared).

## Fixed

- **`portos-server` crash-loop on `Cannot find package 'multer'` from `videoGen.js`.** The new `server/routes/videoGen.js` was written against `multer` for `multipart/form-data` uploads but `multer` was never added to `server/package.json`, so a fresh `npm install --prefix server` left the server unable to import the route — `pm2 logs portos-server --err` showed an `ERR_MODULE_NOT_FOUND` loop and the API never bound to :5555. Fixed by switching to the existing `server/lib/multipart.js` helper (a stream-to-disk multer.diskStorage replacement that was already in the tree), which has the same `req.file = { path, originalname, mimetype, size }` contract. No new dependency added — per `CLAUDE.md`'s "default to writing code yourself", reusing the in-tree replacement is the right call. File: `server/routes/videoGen.js`.

- **`scripts/open-ui-in-browser.js` polled the wrong port when HTTPS was on.** The auto-open script's polling URLs (`/api/system/health`, `/api/browser/health`, `/api/browser/navigate`) were hardcoded to `http://localhost:5555`, but when a Tailscale cert is provisioned `:5555` speaks TLS only — every plain-HTTP request hit a handshake mismatch and timed out. The script computes a `LOCAL_BASE` once at startup: when `data/certs/cert.pem` exists it uses the loopback HTTP mirror (`http://localhost:5553`) for everything; otherwise plain `http://localhost:5555`. Same logic was already correct for the URL handed to the browser; now it's also used for the polls. File: `scripts/open-ui-in-browser.js`.

- **Federation peers stay offline after enabling Tailscale HTTPS.** When `npm run setup:cert` provisioned an HTTPS cert, every outbound federation hop (probe, query, announce, peer Socket.IO relay) started failing because Node's default `fetch()`/socket.io-client run with `rejectUnauthorized: true` and the peer's cert (real Let's Encrypt or self-signed fallback) couldn't always be verified by the calling instance. Per `CLAUDE.md`'s security model the Tailnet IS the trust boundary, so federation traffic doesn't need a second TLS validation layer on top of WireGuard. New `server/lib/peerHttpClient.js` exports a `peerFetch()` that uses `https.request` with a shared `https.Agent({ rejectUnauthorized: false, keepAlive: true })` for `https://` URLs and falls through to global `fetch` for `http://` URLs, plus a `peerSocketOptions` constant (`rejectUnauthorized: false`, `transports: ['websocket', 'polling']`) spread into the `io()` call in `peerSocketRelay.js`. The `peerBaseUrl()` builder is unchanged — the URL string is the same; only the dispatcher tolerates the cert. Two side issues fixed in the same pass: (1) the announce protocol now carries the announcer's Tailscale-issued hostname (`host` field added to `announceSchema` and to `announceSelf`'s POST body, sourced from a new `getSelfHost()` helper that reads `data/certs/meta.json` or `PORTOS_HOST` env override). `handleAnnounce` adopts the host on the receiving side when the peer record doesn't already have one, never overwriting a user-set value. This means bidirectional peer auto-creation finally produces records that can be reached over HTTPS instead of leaving an `address`-only row that can only speak plain HTTP. (2) Probe failures now log a classified one-line diagnostic (`DNS lookup failed`, `Connection refused`, `Host unreachable`, `Probe timeout`, `TLS reject`) instead of a generic `err.message`, so the next time something silently goes offline the cause is visible without strace. (3) `startPolling()` now clears `nextProbeAt`/`consecutiveFailures` on every peer at boot — a server restart is a clear "try again now" signal, and without this peers stuck at backoff tier 5 (24h) wouldn't auto-recover for up to a day after the fix is deployed. Files: `server/lib/peerHttpClient.js` (new), `server/lib/peerSelfHost.js` (new), `server/services/instances.js` (`peerFetch` everywhere, `getSelfHost()` in `announceSelf`, `host` adoption in `handleAnnounce`, `classifyProbeError`), `server/routes/instances.js` (`announceSchema.host`), `server/services/peerSocketRelay.js` (`peerSocketOptions`, `peerFetch` for initial agent fetch). Tests: `server/lib/peerHttpClient.test.js` (new, 2 tests for `peerSocketOptions` shape + `peerFetch` http fall-through), `server/lib/peerSelfHost.test.js` (new, 4 tests for env override / missing meta / tailscale mode / self-signed mode), `server/services/instances.test.js` extended with 4 new `handleAnnounce` host-flow tests.

- **Port story documented and startup banner clarified.** The unified port allocation has been correct in code since the Tailscale HTTPS work landed but docs and startup messages disagreed: `CLAUDE.md` still showed `health: 5554` as an example, the README told users to open `http://localhost:5554`, and the boot logs split the API URL and the loopback URL across two unrelated lines. There's now a single canonical statement of the rules: `:5555` is **always** the user-facing port (its scheme flips between HTTP and HTTPS based on whether `data/certs/{cert,key}.pem` exists, but the port number does not); `:5553` is a loopback-only HTTP mirror that spawns when HTTPS is on so curl/scripts skip cert warnings; `:5554` is dev-only Vite. The startup banner now prints all reachable URLs in one block (`https://<host>.<tailnet>.ts.net:5555`, `https://<tailscale-ip>:5555 (cert warning)`, `http://localhost:5553`), with the trusted hostname URL drawn from `getSelfHost()`. Files: `CLAUDE.md` (Port Allocation section corrected), `docs/PORTS.md` (new "How `:5555`, `:5553`, and `:5554` Relate" section with diagram), `README.md` (Quick Start + Network Access updated), `server/index.js` (banner), `client/vite.config.js` (one-line comment confirming the hardcoded `:5555` is intentional).

## Fixed

- **Image Gen preview no longer flickers between progress and "Waiting for first preview..."** — `useImageGenProgress()` now merges progress events instead of replacing the whole state, so the throttled `currentImage` (server emits a base64 frame every ~2s while polling every 500ms) and any progress event missing `step`/`totalSteps` (some SD API states omit them) no longer cause the preview pane to revert to the empty/waiting state between frames. The hook also seeds initial progress from the `started` event (`step: 0, totalSteps: N`) so the status row shows "Step 0/N" immediately, never "Waiting for first preview..." once a render has begun. Affects both the Image Gen page and the Settings → Image Gen Test Render card. File: `client/src/hooks/useImageGenProgress.js`.

- **Dashboard layout picker dropdown no longer clips off-screen on mobile** — the layout selector menu uses `absolute right-0` to anchor its right edge to the trigger button. On a narrow viewport where the picker sits mid-row (after Server status, before Arrange/Ambient), the 224px-wide menu would extend past the left edge of the screen and cut off layout names ("ything", "us", "ning Review"). `LayoutPicker` now measures the trigger's bounding rect on open: when right-aligning would clip, it switches to `position: fixed` anchored to the viewport's right edge with an 8px margin instead. Added `max-w-[calc(100vw-1rem)]` so the menu can never exceed the viewport regardless of placement, `min-h-[40px]` on every row for the project's 40px touch-target standard, and `shrink-0` on the leading icon + trailing "Built-in" badge so they never collapse when the layout name is long. The dashboard header button row also wraps now (`flex-wrap items-center justify-end gap-2 sm:gap-3`) so on very narrow screens the controls re-flow instead of crowding into a single overflowing row. Files: `client/src/components/dashboard/LayoutPicker.jsx`, `client/src/pages/Dashboard.jsx`.

- **`./setup.sh` no longer leaves a fresh checkout half-installed when native PostgreSQL is chosen.** Two bugs combined to make the script exit "successfully" on a clean machine while the resulting app couldn't boot: (1) `scripts/setup-db.js` treated `pg_isready` as proof that the PortOS database was set up, so when a user already had Homebrew Postgres listening on 5432 and picked **2) Native PostgreSQL** at the prompt (or had `PGMODE=native` in `.env`), the script printed "✅ System PostgreSQL ready on port 5432" and exited 0 — never running `db.sh setup-native`, which is what creates the `portos` role, `portos` database, `vector`/`pgcrypto` extensions, and applies `init-db.sql`. The next `npm start` would then fail to connect with `portos`/`portos`. (2) `setup.sh` ran only the root `npm install` and never installed `client/`, `server/`, or `autofixer/` workspace deps, ran the esbuild postinstall, or rebuilt `node-pty`. The fix replaces those two calls with a single `npm run install:all` (which already does the full install + `npm run setup`) and adds a domain-level readiness probe (`isPortOSDbReady` — tries to authenticate as `portos@portos` and probe the `memories` table) to both native-mode branches in `setup-db.js`: when the probe passes the script exits fast (no startup-time hit on `npm start`/`npm run dev`, no recurring `ALTER USER ... PASSWORD` side effect against the `portos` role); when the probe fails — fresh checkout, missing role, missing schema, wrong password — the script falls through to `db.sh setup-native` to bootstrap. `setup.sh` also gains a final `Start PortOS now via pm2? (Y/n)` prompt (with reprompt on unrecognized input) so a fresh `git clone && ./setup.sh` can take the user from zero to a running pm2 stack on `http://localhost:5555` in one invocation. Files: `scripts/setup-db.js`, `setup.sh`.

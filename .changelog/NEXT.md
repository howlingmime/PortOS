# Next Release

## Added

- **Writers Room (`/writers-room`) — Phase 1 authoring core.** New top-level `Create` sidebar group bundles **Media Gen** (default landing) with **Writers Room**, the new file-backed workspace for prose/short-story/screenplay/novel/essay drafts. Library pane manages folders + works; Markdown editor with live word count, segment-index outline (auto-derived from `#` chapters and `##`/`###` scenes), version snapshots, and Cmd/Ctrl+S save; "Write for 10" sprint timer with 5/10/15/25-min presets, prompt field, and per-session word-delta logging. Storage lives under `data/writers-room/` (`folders.json`, `exercises.json`, `works/<id>/manifest.json` + `drafts/*.md`) — driven by a single writer (`server/services/writersRoom/local.js`) following the Creative Director pattern. New routes under `/api/writers-room/*` cover folder/work CRUD, draft body I/O, version snapshot/switch, and exercise lifecycle. Backend tests cover word counting, content-hash determinism, segment-index splitting, folder/work CRUD with cross-validation (deleting a non-empty folder, missing-parent rejection), draft round-trip + version pointer-switching across multiple snapshots, exercise duration clamping + word-delta math + settled-state rejection, and the strict Zod schemas at the route boundary. Mobile/touch-friendly: library pane caps height on mobile, delete buttons stay reachable without hover. Phases 2-5 (manual AI analysis, Creative Director handoff, synced prose/script/media review, realtime CD feedback) tracked in `docs/features/writers-room.md`.
- **Queue additional renders while one is in flight.** Image Gen form fields stay editable during a render and the Generate button becomes a "Queue" button — submissions land in `mediaJobQueue` (server FIFO) without taking over the active SSE/preview, so the user can stack multiple prompts without waiting. A "+N queued" badge surfaces queue depth, and a 4 s `/api/media-jobs` poll syncs it against the server's truth and refreshes the gallery the moment a queued render completes (only on busy-count drop — no per-tick refresh churn). The poll's effect deps key on a `queueActive` boolean so the interval clock doesn't tear down on every count change. A separate "Cancel current" button lets the user kill the active render without losing the form's content. Mirrors the form-editable-during-render policy already in place for video gen.
- **STAGE pipeline for FLUX.2 model loading + downloads.** `flux2_macos.py` now emits `STAGE:download-tokenizer:<repo>` / `STAGE:download-pipeline:<repo>` / `STAGE:move-to-device` / `STAGE:inference` markers; `server/services/imageGen/local.js#handleLine` parses them and broadcasts a typed `{type:'stage', stage, detail}` SSE event. The Image Gen page maps each marker to a friendly label via a `STAGE_LABELS` lookup ("Downloading model weights (~8 GB on first run)…", "Moving model to GPU…", etc.) so the multi-minute first-run HF weight download no longer looks like a frozen "step 0/8". Inference progress events also carry the active phase so the step counter only ticks during real diffusion, not during download `tqdm` bars (which count safetensors files, not steps).
- **Friendly typed errors for HuggingFace failures.** Top-level exception handler in `flux2_macos.py` walks the exception cause chain (diffusers wraps HF errors in `OSError`) to detect `GatedRepoError` / `HfHubHTTPError(401)` / `RepositoryNotFoundError`, then emits a structured `USER_ERROR:<kind>:<repo>` line plus a friendly `❌ <prose>` line — including a `_repo_from_hf_error()` URL parser that reconstructs `owner/name` from the failing request URL because `huggingface_hub` doesn't always populate `error.repo_id`. `local.js` parses the `USER_ERROR:` line out of the stderr buffer (single `findIndex` pass — no nested `indexOf` O(n²)) and propagates `kind` + `repo` through both the per-job SSE error event and the `imageGenEvents.emit('failed', ...)` payload, so the queue's downstream broadcast inherits the friendly message instead of "Exit code 1". The Image Gen page renders a typed error card with an `ERROR_HEADINGS` lookup ("Model access required" / "HuggingFace token rejected" / "Model repo not found") and a one-click "Request access to <repo> ↗" button that deep-links to the HF page — replacing the previous behavior where a 9B gated-access failure surfaced as bare "Exit code 1" with a wall of Python tracebacks in `pm2 logs`.
- **Python child stderr/stdout echoed to pm2 logs.** `local.js` now logs `🐍 [<jobId>] <line>` / `🐍-out [<jobId>] <line>` for every non-progress, non-noise line from `flux2_macos.py` (and any future image runner). Lines that drove a `progress` or `stage` SSE event are intentionally excluded so the log stream isn't drowned by tqdm bars. Mirrors the existing video-gen logging pattern — without it, model-load lifecycle (`🔧 sdnq: pipeline ← …`) was only visible in the SSE channel the browser saw and never landed in server logs, making "stuck on step 0/8 for 8 minutes" indistinguishable from a hang.
- **In-app HuggingFace token entry.** The "FLUX.2-klein is gated, export HF_TOKEN before running PortOS" banner now has a paste-and-save form: enter your `hf_…` token, click Save, and PortOS persists it to `settings.imageGen.hfToken`. The local-image worker (`server/services/imageGen/local.js`) injects it into the spawn env (`HF_TOKEN`, `HUGGINGFACE_HUB_TOKEN`, `HUGGINGFACEHUB_API_TOKEN` — `huggingface_hub` reads any of the three) so `flux2_macos.py` can pull gated repos like `black-forest-labs/FLUX.2-klein-4B` without the user having to drop to a shell. New helper `server/lib/hfToken.js` (`getHfToken()` / `hfTokenEnv()` / `isPlausibleHfToken()`) — stored token wins over env-var fallback so the user can update without restarting PortOS. New route `POST /api/image-gen/setup/flux2-token` validates the `hf_…` prefix before persisting. Same security trade-off as other PortOS settings: plaintext in `data/settings.json`, justified by the single-user Tailscale deployment model documented in CLAUDE.md.
- **In-app FLUX.2 venv installer.** Selecting a FLUX.2 model when the venv isn't bootstrapped now surfaces an "Install FLUX.2" button on the Image Gen page instead of telling the user to drop to a shell and run `INSTALL_FLUX2=1 bash scripts/setup-image-video.sh`. Clicking it opens a modal that streams real-time progress over a new SSE endpoint (`GET /api/image-gen/setup/flux2-install`) — a 5-stage pipeline (detect Python → create venv → upgrade pip → install packages → verify `Flux2KleinPipeline` import) animates with check-marks/spinners while pip output scrolls in a live log pane below. Closing the modal mid-install SIGTERMs pip so a multi-GB torch download doesn't keep going invisibly. Server-side logic lives in `installFlux2Venv()` in `server/lib/pythonSetup.js`, mirroring the bash script's package list (`FLUX2_PIP_SPECS`) so both paths stay in sync. The status cache (`resolveFlux2Python`) busts on completion so the banner clears without a server restart.
- **FLUX.2-klein image generation on macOS (Apple Silicon).** Three quantized variants ship: 4B SDNQ-4bit (~8 GB @ 512px, recommended), 9B SDNQ-4bit (~12 GB, needs 32+ GB RAM), and 4B Int8 (~16 GB). Runs through PyTorch + diffusers + MPS in a separate `~/.portos/venv-flux2` to avoid colliding with mflux's older torch. `INSTALL_FLUX2=1 bash scripts/setup-image-video.sh` bootstraps the venv (idempotent — skips re-install when `Flux2KleinPipeline` already imports). The runner (`scripts/flux2_macos.py`) mirrors `mflux-generate`'s CLI surface so the existing local-image dispatcher just swaps the binary path; live preview frames, metadata sidecar, gallery, and Remix all work unchanged. Stepwise preview decodes latents on a `@torch.no_grad()` callback and writes capped 512-px PNGs so fs.watch + base64 stays cheap. Models are gated on HuggingFace — accept the license at https://huggingface.co/black-forest-labs/FLUX.2-klein-4B and export `HF_TOKEN`; the Image Gen page surfaces a yellow banner when the venv is missing or the token isn't set. The legacy `flux2-klein-{4b,9b}` registry entries (which had `broken: 'macos'`) auto-upgrade in place — user customizations like a custom name or steps are preserved.

New `/api/media-jobs` lets the UI inspect and cancel queued/running renders.

New `gemma-bf16` (now default, best output quality), `gemma-qat-4bit` (auto-detected if LM Studio installed it), and `gemma-4bit` text-encoder options. Loader resolves `~/...` and prefers a local `localPath` when it exists.

## Fixed

- **FLUX.2 stepwise previews now actually render.** The callback's `vae.decode(latents)` was silently failing every step with `IndexError: tuple index out of range` because `Flux2KleinPipeline` hands the callback latents in transformer-packed shape `(B, num_patches, C·P²)` (e.g. `(1, 2304, 128)` for 1024×576) but the VAE expects unpacked `(B, C, H, W)`. New `_unpack_flux2_latents()` does the patch unpack (2×2 patches × 8× VAE downsample) before the decode call. `make_stepwise_callback()` now takes `height`/`width` and short-circuits the unpack if a future pipeline version hands us an already-4-D tensor.
- **FLUX.2 install detection now checks functional health, not just file existence.** Previously `resolveFlux2Python()` returned the venv binary path as long as the file existed — but a venv whose install was killed mid-pip leaves the python binary behind with an empty site-packages, so the status endpoint reported `venvInstalled: true`, the install banner cleared, and rendering crashed with `Exit code 1` in 100ms when `flux2_macos.py` failed to `import diffusers`. New helper `isFlux2VenvHealthy()` (cached) probes `from diffusers import Flux2KleinPipeline` once. `/setup/flux2-status` and `/setup/flux2-install`'s early-return both use it, so a half-broken venv now reports as not-installed and the user can retry the install from the UI without manual cleanup.
- **PM2 watcher debounce reduced from 5 min → 1 min** and `ignore_watch` hardened with `data/`, `logs/`, `.cache/`, and `portos-stepwise-*` patterns. `watch: ['server']` already scopes the watcher correctly, but the long debounce meant a server-file edit + start-render-now sequence could cleanly kick off a render and then have PM2 restart 5 minutes in, killing the Python child with SIGINT halfway through. Tighter debounce + explicit defense-in-depth ignore patterns make the failure mode less catastrophic and easier to predict during dev.
- **FLUX.2 stepwise preview diagnostic.** Added a one-time `🖼️  stepwise: callback live, latents.shape=…` log on the first callback fire, and a post-run `🖼️  stepwise summary: callback fired N times, saved M previews` so when previews don't show up the cause (callback never attached, latents missing, VAE decode failing) shows up in `pm2 logs`. The existing per-step `⚠️ stepwise preview failed` line now also includes the exception type for triage.

- **Creative Director smoke-test fixture is ~2× faster.** Drops scene `durationSeconds` from 3s → 2s and switches `aspectRatio` from `1:1` (512×512) to a new hidden `1:1-small` preset (384×384, ~44% fewer pixels) — total compute budget goes from `3 × 3s × 512²` to `3 × 2s × 384²`, roughly a 63% pixel-frame reduction. The smaller preset is registered in `ASPECT_PRESETS` only (not `ASPECT_RATIOS`), so the user-facing dropdown and route validation surface stay unchanged; the smoke route bypasses validation by constructing the project internally. `targetDurationSeconds` updated to 6 to match (3 × 2s). Lets the post-pipeline-change health check finish in render time only without waiting on full-resolution clips.
- **Creative Director agent prompts moved into the Prompts Manager.** `buildTreatmentPrompt` and `buildEvaluatePrompt` (in `server/lib/creativeDirectorPrompts.js`) now compute a flat view object and delegate to `buildPrompt('cd-treatment', view)` / `buildPrompt('cd-evaluate', view)`. The actual prompt text lives in `data/prompts/stages/cd-treatment.md` and `cd-evaluate.md` — editable from the Prompts Manager UI without code changes. The view object precomputes everything the template engine can't (aspect dims, render params, multi-frame list with timeline-position labels, scene strategy line, retry budget delta), so authors can rearrange wording and sections freely. `agentBridge.js` updated to await the now-async builders. The two new stages plus their templates ship as starters in `data.sample/prompts/stages/cd-treatment.md`, `cd-evaluate.md`, and the `cd-*` entries in `data.sample/prompts/stage-config.json` — `npm run setup` / `npm run setup:data` copies them into `data/prompts/` on fresh installs (and fills in any missing files on existing installs without overwriting user customizations).
- **Prompt template engine upgraded to support dot notation.** `server/lib/promptTemplate.js` ships an enhanced Mustache-flavored renderer that handles `{{a.b.c}}` lookups, object section scopes (`{{#user}}{{first}}{{/user}}`), array iteration with parent-scope fall-through (so `{{project.id}}` still resolves inside `{{#frames}}`), `{{.}}` for primitive iteration items, inverted sections (`{{^x}}`), and triple-mustache (`{{{x}}}`). Routed through `server/services/promptService.js#buildPrompt` so every PortOS template (including legacy ones like `cos-evaluate.md` that were silently leaving `{{metadata.context}}` literals in the output because the upstream toolkit's `\w+`-only regex doesn't match dots) now renders correctly. Storage stays in the toolkit (`stage-config.json`, `variables.json`, `.md` files); only the templating step is local. 17 engine tests + 12 CD prompt-rendering tests cover the surface.
- **Creative Director multi-frame evaluator sampling.** The cognitive evaluator previously only Read the rendered video's frame-0 thumbnail, so 5s i2v scenes whose intent develops mid- or late-clip (archway appearing, light bloom, particles converging) were getting rejected even when the render delivered. The server now samples 5 evenly-spaced frames at render-completion time via a new `extractEvaluationFrames()` helper in `server/lib/ffmpeg.js` (probes total frames with `ffprobe`, then uses `ffmpeg -vf "select='eq(n,0)+eq(n,N/4)+eq(n,N/2)+eq(n,3*N/4)+eq(n,N-1)'" -vsync vfr`), writes them as `<jobId>-f1.jpg ... -f5.jpg` in `data/video-thumbnails/`, persists the basenames to `scene.evaluationFrames`, and `buildEvaluatePrompt` now lists every frame with timeline position tags ("start (0%)", "~50% through", "end (~100%)") and instructs the agent to Read each one. Includes explicit "intent that arrives late still counts as delivered" guidance to fix the false-reject pattern observed on the first real E2E run. Falls back to single-thumbnail Read when extraction fails (ffmpeg/ffprobe missing, probe returns 0 frames).
- **Creative Director stuck projects now self-recover** — `completionHook` always re-evaluates project state after any CD task completes (regardless of task kind), so legacy `scene`-kind tasks from before the treatment/evaluate split no longer leave a project frozen. PATCHing a scene to a terminal status (`accepted` or `failed`) also nudges the orchestrator to start the next scene immediately.
- **Creative Director projects auto-resume on server restart.** New `recoverInFlightProjects()` runs at boot (after `initMediaJobQueue` reclassifies running jobs as `failed (interrupted by restart)`): for every project in `planning`/`rendering`/`stitching`, it resets any scenes stuck in `rendering` or `evaluating` (whose in-memory listeners died with the previous process) back to `pending` and fires `advanceAfterSceneSettled` so the orchestrator picks up where it left off. Without this, a pm2 reload mid-render left the project frozen — scene status pointed at a dead job, no listener to advance.
- **Creative Director smoke-test fixture** — `POST /api/creative-director/smoke-test` (or the "Run smoke test" button on the CD list page) creates a deterministic 3-scene colored-ball project: scene 1 generates a red ball bouncing on white (text-to-video), scene 2 continues from scene 1's last frame with a "blue ball" prompt to verify the i2v honors a prompt-driven color change while preserving motion, scene 3 continues from scene 2 with the same prompt to verify pure i2v continuity. The fixture sets `disableAudio: true` (skips the mlx_video audio pass to roughly halve wall-clock per scene) and `autoAcceptScenes: true` (sceneRunner short-circuits the cognitive evaluator with a synthetic `accepted` evaluation, so a smoke run completes in render time only — no Claude in the loop). Adds `disableAudio` and `autoAcceptScenes` as CD project fields and plumbs them through to the queue params.
- **Creative Director scene-retry path unblocked.** `creativeDirectorSceneUpdateSchema` previously stripped `pending` from the agent-callable status enum, but the evaluator-agent prompt template (`creativeDirectorPrompts.js`) and `advanceAfterSceneSettled` (`completionHook.js`) both depend on the agent flipping a scene back to `pending` (with a refined prompt and bumped `retryCount`) to request a re-render. The validation rejected every retry PATCH with `Invalid enum value` and stranded scenes in `evaluating` forever. Schema now accepts the full `SCENE_STATUSES` set and the misleading "agents shouldn't be allowed to flip back to pending" comment is corrected.

- **Image-to-video on macOS now actually preserves the source image as the first frame.** Root cause: PortOS was on `mlx-video-with-audio` 0.1.33, which had a known bug where the VAE encoder failed to load for split-format unified models (LTX-2.3 distilled-Q4) and silently produced gray-fog conditioned frames. Pinned `>=0.1.35` in `scripts/setup-image-video.sh`. The setup script also installed the wrong PyPI package: `mlx_video` (a different unrelated "video loading utilities" package) instead of `mlx-video-with-audio` (the LTX generation backend). Both provide an `import mlx_video` module so the wrong install was silent. Verified end-to-end through the API: pixel-perfect first-frame preservation with coherent motion across all frames.
- **FFLF last-frame conditioning silently produced no output.** PortOS was passing `numFrames - 1` (= 24 for 25 frames) as `--image-frame-idx`, but mlx_video expects a *latent* frame index — for LTX that's `floor((videoFrames - 1) / 8)`, max 3 for 25 frames. ffmpeg-conditioning shape check failed and the python child exited 0 with no output mp4. Now computes `lastLatentIdx` correctly.
- **"Continue" / extract-last-frame produced phantom-success logs but no file**, leaving a broken image preview in Video Gen. Two compounded bugs: (1) `-sseof -0.1` was too narrow a seek window — for videos with audio (B-frames + AV mux), the last keyframe sits earlier than 100ms before EOF, so ffmpeg returned zero frames while sometimes still exiting 0. Widened to `-sseof -1.0` and added `-update 1`. (2) The success path resolved on exit-code without verifying the output file existed. Now requires `existsSync(framePath)` too.
- **Last-frame extraction now caches** to `data/images/lastframe-<videoId>.png` — repeated "Continue" clicks reuse the cached PNG instead of re-spawning ffmpeg. Cuts ~1–2s per click.
- **Delete operations now log** for both image (`🗑️ Deleted image: <filename>`) and video (`🗑️ Deleted video: <filename>`) gallery actions. Previously delete was silent, making it impossible to confirm from the server logs that the click hit the backend.
- **Media model registry tolerates partial user edits.** `loadMediaModels()` now merges parsed `data/media-models.json` over the built-in defaults so a registry missing top-level keys (e.g. `{}` or no `video`) doesn't crash server boot at module import. The loader also catches read errors (permissions, broken symlink) — not just JSON parse — falling back to defaults so a registry I/O problem can't take down startup. Wrong-type fields are coerced back to defaults too (e.g. `image: {}` or `video.macos: "ltx"` → built-in array), so consumers like `getImageModels()` / `buildAppModels()` can't throw at import-time on parseable-but-wrong-shape JSON. `getDefaultVideoModelId()` validates the configured default against the platform's available list — typos or `broken`-flagged ids fall back to the first available model with a warning instead of surfacing as "Unknown video model" in the UI. `getTextEncoderRepo()` validates that the resolved entry has a non-empty repo string before passing it to mlx_video. macOS video gen now also validates `model.repo` exists before spawning, surfacing a 500 with a clear error rather than passing `undefined` into spawn args.
- **Last-frame cache validates non-zero size.** A prior ffmpeg crash could leave a 0-byte placeholder at `data/images/lastframe-<videoId>.png`, and the cache hit (`existsSync(framePath)`) would serve it forever as a broken image. `extractLastFrame` now `statSync`s the cached file and falls through to re-extract when it's missing or zero bytes; failed re-extractions also unlink the partial file so the next call retries instead of inheriting the broken cache.
- **Faststart remux works on Windows without risking the rendered video.** `optimizeForStreaming()` did an in-place `fs.rename(tmpPath, videoPath)` which fails on Windows when the destination already exists. The Windows replace strategy now moves the original aside to a `.bak.<uuid>` first, installs the optimized file, and restores the backup on any failure — so a transient rename failure (file lock, AV scan, permissions) skips faststart instead of destroying the rendered video. POSIX still uses the single-syscall atomic replace.
- **`caffeinate` sleep-prevention is now strictly macOS-only.** Previous gate (`!IS_WIN`) would also fire on Linux and emit a pointless ENOENT every render. Tightened to `process.platform === 'darwin'`.
- **`scripts/setup-image-video.sh` removes the wrong package before install.** The PyPI package `mlx_video` (unrelated video-loading utility) shadows the LTX backend's `import mlx_video` if previously installed. The script now `pip uninstall --yes mlx_video` before installing `mlx-video-with-audio>=0.1.35`, so upgraders aren't left in an ambiguous state.
- **Form fields stay editable during a video render.** Previously every input was `disabled={generating}`, blocking the user from queueing a different config while a render was in flight. The queue worker captures payloads at enqueue time, so live editing is safe.
- **"Add to collection" popover on media cards no longer gets clipped at the card edge.** `MediaCard`'s outer wrapper had `overflow-hidden` (to clip the image into the card's `rounded-xl`), which was creating a clipping context that swallowed the `AddToCollectionMenu` popover regardless of its `z-index`. Moved the corner-clipping onto the image button itself (`rounded-t-xl overflow-hidden`) so the outer card no longer clips its absolutely-positioned children.

- **Morse Send drill** no longer reveals the answer or competes with itself for spacebar input: the morse pattern under each prompt is gone (the reference tabs on the right are still available for hints), the global spacebar keying listener only attaches in Send mode (so it doesn't fire on the trainer landing page or during Copy drills), the side widget's "Practice Key" panel is hidden outside Send mode and no longer duplicates the live decoded letter, and the "Your sending" box starts blank with a "waiting for your first key…" hint instead of a misleading "—" placeholder.
- **CoS agents now reliably open PRs and request Copilot reviews when configured.** Previously, when both `openPR` and `reviewLoop` were enabled on a task, the worktree-cleanup logic's gate (`taskOpenPR && !taskReviewLoop`) skipped PR creation — falling through to auto-merge the agent's branch into main with no PR, no review, and contradictory wording in the agent prompt about whether it should open the PR itself. The cleanup now always opens the PR when `openPR` is set; when `reviewLoop` is also set, it requests a Copilot review on the new PR via the new `git.requestCopilotReview()` helper. The agent prompt is updated to consistently tell the agent the system handles PR creation and review requests.

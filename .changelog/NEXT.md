## Added

- **Blood Pressure tracking in MeatSpace ‚Üí Body** ‚Äî feature parity with the MortalLoom iOS/macOS companion app. New Blood Pressure card on the Body tab shows a systolic/diastolic trend chart (with AHA/ACC 2017 guideline reference lines), the latest reading, and its category + estimated life-expectancy impact. Readings are date-upserted so a HealthKit sync on the phone and a manual entry in PortOS on the same day merge into one row. Storage routes through the new `mlUpsertHealthMetricByDate` into MortalLoom's `healthMetrics[]` array when iCloud sync is enabled (same `bloodPressureSystolic`/`bloodPressureDiastolic` keys the Swift app writes), else falls back to `data/meatspace/health-metrics.json`. New endpoints: `GET /api/meatspace/blood-pressure`, `POST /api/meatspace/blood-pressure`. Apple Health XML ingest now maps `HKQuantityTypeIdentifierBloodPressureSystolic`/`Diastolic` ‚Üí `blood_pressure_systolic`/`diastolic`, and those metrics are registered in the Core Vitals section of the Health tab so exported HK data shows trend lines there too. BP classification (normal / elevated / stage-1 / stage-2 / crisis) is ported from `CardioFitnessEngine.classifyBP` into `client/src/components/meatspace/bpClassification.js` so both sides report identical categories for the same reading.
- **Hide voice widget per-browser** ‚Äî the floating voice widget can now be dismissed entirely by clicking the new eye-off icon (next to the mobile FAB or in the desktop controls bar). The preference persists in `localStorage` under `portos.voice.hidden`, so iPhones/Safari sessions where the mic APIs don't work can stay clean while the desktop keeps the widget. Re-enable via **Settings ‚Üí Voice ‚Üí "Show floating voice widget"**; a custom `portos:voice:visibility` DOM event syncs the change live without a reload.
- **Daily Log ‚Äî voice-dictatable diary in the Brain section** ‚Äî new `Brain ‚Üí Daily Log` tab at `/brain/daily-log` for free-form typed or dictated daily entries. One entry per date, stored in `data/brain/journals.json` with per-segment source metadata (`text` vs `voice`). Five voice tools land in the CoS: `daily_log_open` ("open my daily log" ‚Äî navigates + optionally starts dictation), `daily_log_start_dictation` / `daily_log_stop_dictation`, `daily_log_append` (one-shot append without dictation mode), and `daily_log_read` (reads the entry verbatim via TTS). Dictation mode short-circuits the LLM at the server ‚Äî every voice utterance appends straight to the log until the user says "stop dictation" ‚Äî so the CoS stops talking back while you ramble. Tools can now emit client-side side effects (`voice:navigate`, `voice:dictation`); the Voice widget listens and routes the browser + surfaces a "Dictating to Daily Log" chip. Optional Obsidian mirror: each entry auto-writes to `<vault>/<folder>/<YYYY-MM-DD>.md` when a vault is configured in the tab's settings panel (reuses the existing Notes vault registry), picking up iCloud backup for free. Entries flow through `brainMemoryBridge.js` (new `journals` brain type, `daily-log` category) so the CoS memory system vector-embeds each day and agents can semantically search across the log history. New routes: `GET/PUT/DELETE /api/brain/daily-log/:date`, `POST /api/brain/daily-log/:date/append`, `GET/PUT /api/brain/daily-log/settings`, `POST /api/brain/daily-log/sync-obsidian`.
- **Voice CoS tool expansion ‚Äî round two** ‚Äî nine more hands-free tools registered in `server/services/voice/tools.js`, further expanding the Voice CoS toolset:
  - `brain_list_recent` ‚Äî "Read me my last five captures"; returns the newest inbox entries with date + text.
  - `meatspace_log_weight` ‚Äî "Log my weight at 180"; accepts `lb` (default) or `kg`, auto-converts to lb before persisting, defaults to today.
  - `goal_list` ‚Äî "What are my goals?"; lists active goals ordered by urgency with horizon, category, and progress percent.
  - `goal_update_progress` ‚Äî "Bump my jacket goal to 40 percent"; fuzzy-matches the title (exact ‚Üí substring ‚Üí per-word token score), reports previous ‚Üí new percent and flags ambiguous matches.
  - `goal_log_note` ‚Äî "Add a note to my estate goal ‚Äî found the property"; attaches a free-form progress entry to the matched goal without changing the percent.
  - `pm2_status` ‚Äî "Is anything crashed?"; reports online/total plus any process in an unhealthy state with its restart count.
  - `pm2_restart` ‚Äî "Restart portos-api"; restart-only by design (no kill/delete, per the shared-server CLAUDE.md rule), accepts a substring match on process name.
  - `feeds_digest` ‚Äî "What's new in my feeds?"; returns total unread plus the top 5 newest items with title + feed name.
  - `time_now` ‚Äî "What day is it?"; returns ISO + day of week + localized date/time. Included because LLMs don't know current time on their own and frequently hallucinate it.
  All new tools reuse existing PortOS services (`getInboxLog`, `addBodyEntry`, `getGoals`/`updateGoalProgress`/`addProgressEntry`, `listProcesses`/`restartApp`, `getItems`/`getFeeds`) and stay read/write-aligned with the web UI paths ‚Äî no parallel data model. Settings ‚Üí Voice copy updated to reflect the broader tool surface ("brain capture, goal updates, PM2 control, feed digests, time, and more").
- **Voice CoS tool expansion ‚Äî round one** ‚Äî four new hands-free tools registered in `server/services/voice/tools.js` on top of the existing `brain_capture`:
  - `brain_search` ‚Äî substring-search the brain inbox ("what did I say about that book?"); returns up to 5 hits with date + text.
  - `meatspace_log_drink` ‚Äî "I had a beer" / "log two glasses of Cabernet"; known categories (beer/wine/whiskey/shot/cocktail) auto-fill oz+ABV defaults, free-form names also work.
  - `meatspace_log_nicotine` ‚Äî "I had a cigarette" / "two pouches"; presets for cigarette/vape/pouch.
  - `meatspace_summary_today` ‚Äî "How am I doing today?" ‚Äî reports today's drinks + nicotine against 7-day rolling averages.
  All new tools reuse existing services (`captureThought`, `getInboxLog`, `logDrink`, `logNicotine`, `getAlcoholSummary`, `getNicotineSummary`) so behavior stays aligned with the UI paths.
- **Curated Piper voice catalog** ‚Äî Settings ‚Üí Voice now surfaces a hand-picked set of Piper voices for comparison against Kokoro: British (Jenny Dioco, Cori-high, Southern English female), Scottish (Alba), Irish (VCTK p266 via `speakerId: 39`), and an American baseline (Amy). Hits `POST /api/voice/test` with an optional `voice` override that temporarily swaps the voice for the test synth call ‚Äî the saved config is untouched. ## Changed

- **Default STT engine flipped from Whisper to Web Speech API** ‚Äî fresh installs and the sample `data.sample/settings.json` now default to `stt.engine: 'web-speech'` (browser-native, zero-latency, no server process). `reconcile()` in `server/services/voice/bootstrap.js` now short-circuits when STT is Web Speech: stops any leftover `portos-whisper` PM2 app and skips binary/model verification and download so users who stick with the default don't pay for the whisper.cpp model or process. Health probes skip the whisper HTTP check in this mode and report a synthetic `web-speech: { ok: true, state: 'browser-native' }` status so the Settings page no longer shows whisper as "down" when it's intentionally off. Existing installs keep their saved `stt.engine` value ‚Äî flip it via Settings ‚Üí Voice ‚Üí STT Engine.

- **Web Speech API STT mode for voice assistant** ‚Äî new `stt.engine: 'web-speech'` config option uses the browser's native SpeechRecognition API for speech-to-text instead of whisper.cpp. Transcription runs entirely in-browser with zero network latency ‚Äî final transcripts are sent via the existing `voice:text` path so no server changes needed. Hands-free toggle and VAD meter hidden in this mode (browser handles VAD natively). Interim transcripts appear live in the conversation panel. Enable via Settings ‚Üí Voice ‚Üí STT Engine.
- **Local voice Chief-of-Staff with hands-free VAD and opt-in tool calling** ‚Äî new voice stack under `server/services/voice/` and `client/src/{services,components}/voice/` runs entirely on-device: Whisper (STT) + Kokoro or Piper (TTS) + LM Studio (LLM). Two capture modes: push-to-talk (click once to listen, click again to send) and **hands-free continuous mode** (default) that auto-submits on a ~600ms pause and lets you barge in over the bot's own voice. Hands-free is built around an AudioWorklet streaming raw PCM to a thin energy-based VAD ‚Äî thresholds auto-calibrate to the user's ambient noise floor on startup, a pre-roll ring buffer avoids clipping the first syllable, a TTS echo-tail cooldown filters speaker‚Üímic reverb, and a runaway-speech watchdog unjams VAD if silence never registers. Whisper requests now bias the decoder with a PortOS vocabulary prompt so proper nouns ("brain inbox", "Chief of Staff", "Tailscale") don't drift to homophones.
- **Configurable CoS personality for the voice assistant** ‚Äî Settings ‚Üí Voice exposes a personality editor (name, role, traits, speech style, custom prompt). The system prompt is composed live per turn from those fields plus the list of available tools. Toggle **Enable tools** (off by default; needs a tool-use-trained model like Qwen2.5-7B or Hermes-3) to let the CoS actually do things. First tool shipped: `brain_capture` ‚Äî routes any "remember‚Ä¶", "add‚Ä¶", "note‚Ä¶" style request into the existing brain inbox via `captureThought()`. The pipeline runs up to 3 tool-call rounds per turn and streams text deltas to TTS between iterations so you hear the CoS narrate what it's doing.
- **Voice widget on every page** ‚Äî floating conversation card (bottom-right) with live mic-level meter, stage indicator (`handsfree` / `capturing` / `thinking` / `speaking`), mode toggle, interrupt button, text-entry fallback, and scrolling transcript. Hotkey (default Space) toggles listening. Settings ‚Üí Voice also gets STT/TTS engine pickers, Whisper model picker, Kokoro voice picker (dozens of voices: `af_*` female, `am_*` male, `bf_*`/`bm_*` British), CoreML encoder toggle, and one-click "Test voice".
- **MortalLoom iCloud sync** ‚Äî new Settings tab (Settings ‚Üí MortalLoom) lets PortOS share Goals and Meatspace data with the MortalLoom iOS/macOS app by reading from and writing to the app's iCloud ubiquity container (`~/Library/Mobile Documents/iCloud~net~shadowpuppet~MeatSpaceTracker/Documents/MortalLoom.json`). When enabled, PortOS treats the iCloud file as the source of truth for alcohol drinks, nicotine entries, blood tests, body composition, epigenetic tests, eye exams, and goals ‚Äî and every add/update/remove flow mirrors back to it so edits on either side stay in sync. Non-destructive one-shot "Import from MortalLoom" button for adopting the iCloud data into local PortOS stores by id-dedup. New `server/services/mortalLoomStore.js` exposes tight primitives (`readStore`, `updateStore`, `mlPush/mlPatchById/mlRemoveById/mlReplace`, `mlArrayIfEnabled`, `mlIdAtDateIndex`) used by `meatspaceHealth`, `meatspaceAlcohol`, `meatspaceNicotine`, and `identity`.
- **MortalLoom promo banner on Meatspace + Goals pages** ‚Äî dismissible banner (stored in `localStorage` as `mortalloom-banner-dismissed-v1`) linking to the App Store (`id6760883701`) and to the new Settings tab for iCloud sync configuration.

## Fixed

- **Voice agent silent after every question when `model: 'auto'`** ‚Äî the Chief-of-Staff transcribed the user's speech but never replied (server log showed `ttft=‚Äîms ¬∑ tts=0ms (0 sentences)` with no error). Two root causes: (1) `'auto'` resolved to whichever LM Studio model had the smallest `B` suffix in its id ‚Äî on IBM Granite 3.x installs that is Granite itself, which does not emit OpenAI-format `delta.tool_calls`; it emits its own `<tool_call>[{...}]</tool_call>` (and sometimes `<tool_request>[{...}]`, often unclosed) inline in `delta.content`. With the full 19-tool schema attached, Granite silently bailed with a single empty SSE chunk and closed the stream. (2) The pipeline only logged a turn-summary at the END, so a 60s reasoning loop looked like the server was dead. Fixes:
  - `resolveModel` now prefers a curated tool-use allowlist (Qwen2.5-Instruct, Hermes-3, Mistral-Small, Devstral, Ministral, Llama-3.1+-Instruct, Command-R, Functionary) over size-ranked picks when `tools.enabled`, and steps past known-incompatible families (Granite, tiny non-instruct Gemmas).
  - `extractInlineToolCalls` parses both `<tool_call>` and `<tool_request>` spellings in either closed or unclosed form and hoists them into structured `toolCalls`, with a balanced-bracket JSON scanner that survives `]` characters inside string arguments. A matching streaming stripper withholds those fragments from `onDelta` so raw JSON never reaches TTS.
  - `ensureToolCapableModel` auto-installs a default tool-capable model the first time voice reconciles with no compatible model present ‚Äî shells out to `lms get -y lmstudio-community/Qwen2.5-7B-Instruct-GGUF` (overridable via `PORTOS_VOICE_DEFAULT_TOOL_MODEL`). Runs fire-and-forget so whisper/TTS come up immediately.
  - Per-event timing logs with a turn correlation id: `turn.start`, `stt.done`, `lmstudio.resolve`, `lmstudio.headers`, `llm.start`, `llm.ttft`, `llm.done`, `tool.start`/`tool.done`/`tool.fail`, `tts.start`/`tts.done`, `turn.summary`. Makes it obvious whether latency is in cold-load, reasoning, tool execution, or TTS.
  - Empty-output guard ‚Äî when the LLM produces no text and no tool runs, the server now emits `voice:error` with a specific model-switch hint (the existing VoiceWidget toast surfaces it) instead of going silent.

- **Voice widget covering page content on mobile** ‚Äî the floating voice input bar (text field, hands-free toggle, mic button) was fixed at the bottom of the screen and spanned the full width on mobile, blocking form buttons (Cancel/Save) and navigation. On mobile (<768px) the widget now collapses to a single compact mic FAB in the bottom-right corner; tapping it expands the full controls and starts listening in one tap. An X button dismisses back to the FAB. Desktop behavior is unchanged.

- **Cmd prompt windows flashing on Windows** ‚Äî `fetchJlist` in `pm2.js` was spawning a `node pm2.js jlist` subprocess every 400ms+ during app-status polling. Even with `windowsHide: true`, Windows briefly flashes the console for spawned Node.js processes on some configurations. Fixed by switching to the PM2 Node.js API (`pm2.list()` via IPC) for the default PM2_HOME case, eliminating subprocess spawning entirely. The CLI path is kept only for custom `pm2Home` (managed external apps).
- **Digital Twin build errors ‚Äî missing Soul-prefixed API exports** ‚Äî components used `api.createSoulDocument`, `api.runSoulTests`, etc. but `apiDigitalTwin.js` only exported `DigitalTwin`-prefixed names. Added 10 Soul aliases so the build resolves all imports.

- **POST drill scoring paired every answer to the first prompt** ‚Äî `llmResponseSchema` in `postValidation.js` didn't list `questionIndex`, so Zod's default strip mode dropped the field on its way to the scorer. The scorer's `r.questionIndex ?? i` fallback then resolved to the response-array index ‚Äî always 0 for the single-response submits the trainer makes ‚Äî and every answer was evaluated against the FIRST prompt of the drill. Affected bridge-word, idiom-twist, double-meaning, and every other LLM-scored POST drill (the v1.43.0 client-side fix wired up `questionIndex` on the request but it never reached the scorer). Schema now declares `questionIndex` and a regression test in `postValidation.test.js` covers the round-trip.
- **Compound chain rejected the natural shorthand** ‚Äî typing just the *other half* of a compound (e.g., `hose`, `pit`, `work` for the root `fire`) was scored as invalid because the local scorer required the user's input to literally include the root word. Players had to type `firehose` / `firepit` / `firework` in full, which slowed them down and produced misleading "0 valid compounds" scores. Scorer now accepts both the full compound and the half ‚Äî bare root word still rejected, and the "you could also have said" suggestions correctly suppress examples the user already covered via either form. UI placeholder updated to "Type the full compound or just the other half...".
- **Brain inbox classification with Gemini CLI was slow** ‚Äî the provider had no default model, so `gemini` fell back to `gemini-2.5-pro` (a thinking model) and was also invoked without `--prompt`, forcing interactive-mode detection on every classify call. `brain.js callAI()` now sends gemini-cli prompts via `--prompt`, passes `--output-format text`, and defaults to `gemini-2.5-flash-lite` (thinking disabled) when no model is set. Sample providers config populates `models`/`defaultModel`/`lightModel`/`mediumModel`/`heavyModel` for gemini-cli so new installs pick up the fast model automatically.

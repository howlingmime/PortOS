# Next Release

## Added

- **One-click "Enable HTTPS" button on the Instances tab.** The Tailnet DNS & trusted HTTPS explainer used to instruct users to drop into a shell and run `npm run setup:cert`. New button on the same banner runs the same flow at runtime via a new `POST /api/instances/provision-cert` endpoint, which calls `tailscale cert` for the local MagicDNS hostname, writes `data/certs/{cert,key}.pem`, and updates `meta.json`. Failure modes (Tailscale not installed, not running, no MagicDNS hostname, HTTPS Certificates feature disabled in the tailnet admin) surface as toasts with actionable guidance. Because the HTTPS listener type is decided at server boot in `lib/tailscale-https.js`, the response includes `requiresRestart: true` whenever PortOS is currently running over HTTP â€” the UI then prompts the user to restart so the new cert actually takes effect; if HTTPS was already running, the existing `certRenewer` hot-swap path picks up the renewed cert without a restart. The button is disabled until MagicDNS is detected, mirroring what the underlying CLI requires. Files: `server/services/certProvisioner.js` (new), `server/routes/instances.js`, `client/src/services/apiSystem.js`, `client/src/pages/Instances.jsx`.
- **Image Gen â€” third backend: Codex CLI `$imagegen`.** Image Gen can now route generation through the user's locally-installed Codex CLI built-in `image_gen` tool â€” invoked headlessly with `codex exec --skip-git-repo-check --sandbox workspace-write '$imagegen <prompt>'` so it uses the user's logged-in Codex session (no `OPENAI_API_KEY` required). New provider at `server/services/imageGen/codex.js` mirrors `local.js` (single-job model, SSE progress events) but harvests the resulting PNG from `~/.codex/generated_images/<session-id>/ig_*.png` after parsing the session id out of the codex launch banner â€” codex's stdout has no machine-readable image path, so banner-parse + dir-poll is the contract. Codex usage is gated by an explicit `Enable Codex Imagegen` setting because not every Codex plan exposes the `image_gen` tool â€” when disabled the option disappears from settings tile, image-gen page chip, voice tool dispatch, and palette gating up front.
- **Image Gen â€” per-render backend chip.** New chip strip on the Image Gen page lets the user flip between Local / External / Codex per-render without changing the saved default. Only viable backends (External URL set, Local Python set, Codex enabled) appear; the strip hides itself when only one backend is configured. Selection is sent as `mode:` in `POST /api/image-gen/generate` and the dispatcher honors the override. Codex mode also hides backend-irrelevant knobs (steps, seed, guidance/CFG, LoRAs, quantize) since the built-in `image_gen` tool decides those internally. `GET /api/image-gen/status?mode=codex` lets the page probe a non-default backend before saving.

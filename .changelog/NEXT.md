# Next Release

## Fixed

- **Reverting a peer to plain IP (after enabling Tailnet DNS) is now sticky.** When a user cleared a peer's `host` on the Instances page to switch the federation hop back to `http://<ip>`, the next inbound `peers/announce` from that peer would silently re-adopt the DNS name — the only safeguard was "is `existing.host` empty?", which couldn't distinguish "user explicitly cleared" from "never set yet." Added a `hostManual` flag on each peer record: latched to `true` whenever the user sets or clears the host via `PUT /api/instances/peers/:id`, and respected by `handleAnnounce` (auto-adoption is gated on `!existing.host && !existing.hostManual`). Fresh peers and peers that were auto-learned still receive automatic host updates from announces; only explicit user intervention pins the choice. Also added a one-click `use IP only` inline action on each peer's host editor and a discoverable banner at the top of the Instances page that summarizes Tailnet DNS state, surfaces the local instance's MagicDNS name in the SelfCard, and walks through the three-step setup (MagicDNS admin toggle → `npm run setup:cert` → click `use <host>` per peer). Files: `server/services/instances.js`, `server/services/instances.test.js`, `client/src/pages/Instances.jsx`.
- **LM Studio auto-loads a model on "No models loaded" 400.** Any caller of `callProviderAISimple` (taste questionnaire, identity, autobiography, insights, jiraReports, goalCheckIn) that hit a cold LM Studio used to error out with `Provider returned 400: { ... "No models loaded" ... }`. Now on that specific error pattern, the helper lists LM Studio's downloaded LLMs via `/api/v0/models`, picks one (preferring `provider.defaultModel`, then any id in `provider.models` that's downloaded, else the first downloaded LLM), loads it via `POST /api/v1/models/load`, and retries the chat-completion. Mirrors the existing pattern in `services/memoryClassifier.js` but driven from the provider config so every API caller benefits. File: `server/lib/aiProvider.js`.
- **Stop showing the same error toast 3-4 times.** A single failing API call could surface its error message multiple times because three independent channels all toasted the same payload: `apiCore.request` toasts on every non-OK response, the server emits both `io.emit('error:occurred')` (broadcast) and `errorEvents → broadcastToErrors('error:notified')` (subscriber) for the same error, and `useErrorNotifications` listened to *both* socket events with the same handler. Two changes: (1) `useErrorNotifications` now subscribes to the targeted `error:notified` channel only — `error:occurred` was the duplicate broadcast leg — and (2) the Toast layer added a content-based dedup window (1.5s) so any duplicate from any source is silently dropped. Explicit-`id` toasts (loading→success swaps) skip dedup so progress toasts still update in place. Files: `client/src/components/ui/Toast.jsx`, `client/src/hooks/useErrorNotifications.js`.
- **Media Gen Video defaults to the smallest model.** The macOS default was LTX-2 Unified (~42 GB). New default is LTX-2.3 Distilled Q4 (~22 GB) so a fresh install fits on smaller machines without a manual override. File: `server/services/videoGen/local.js`.
- **Taste questionnaire falls back to a configured API provider.** The Digital Twin → Taste tab was auto-selecting the first enabled provider for `Generate Summary` / `Go Deeper`. When that happened to be a CLI provider (Claude Code, Codex, Gemini CLI), the request failed with a generic "requires an API-based provider" toast — three times, because both the API client and two socket subscribers re-toasted the same error. Now (1) the section provider picker filters to API-type providers only, (2) the server resolves any usable API provider when the requested one isn't compatible (requested → active → first enabled API) via a new `resolveAPIProvider` helper in `server/lib/aiProvider.js`, (3) the section-complete view shows an inline link to AI Providers when no compatible provider is configured instead of leaving the user with a wall of red toasts and no direction, and (4) the redundant component-level `toast.error` calls were removed (apiCore already toasts on non-OK responses). Files: `server/lib/aiProvider.js`, `server/services/taste-questionnaire.js`, `client/src/components/digital-twin/tabs/TasteTab.jsx`.

## Changed

- **Send to Video carries the source image's prompts.** Clicking "Send to Video" on an Image Gen result, gallery card, or lightbox — or the same action from Media History — now pre-populates the Video Gen prompt and negative prompt from the source image's metadata. Previously only the source image filename was passed, leaving the user to retype or paste the prompt to keep the same scene direction. The flow piggybacks on the existing `?sourceImageFile=` query-param hand-off: `?prompt=` and `?negativePrompt=` are added when the source has them, and `VideoGen.jsx` reads both on first render and re-syncs on subsequent navigations (mirroring the existing source-image effect). Files: `client/src/pages/ImageGen.jsx`, `client/src/pages/MediaHistory.jsx`, `client/src/pages/VideoGen.jsx`.

## Fixed

- **Voice mode works on Windows.** The voice setup script and TTS engine now run correctly on Windows: `bootstrap.js` invokes `pwsh` with a PowerShell setup script (`scripts/setup-voice.ps1`) instead of the macOS-only bash/Homebrew script; `which` uses `where` on Windows; `verifyBinaries` checks the local piper install path first (skipping a wasteful PATH subprocess when the binary is present); `tts-piper.js` writes piper output to a temp file on Windows to avoid CR+LF corruption of binary WAV data from text-mode stdout. Shared platform constants (`IS_WIN`, `PIPER_BIN_NAME`) extracted to `config.js`.
